PBI-011 — Administrar categorías y relación producto-categoría

Qué se implementó:
Se crearon los endpoints REST para gestión completa de categorías y su relación con productos,
utilizando la tabla puente Producto_Categoria ya definida en el esquema.

Archivos modificados:
- backend/src/api.js

Endpoints creados:
- GET    /api/categorias                              — Lista categorías (solo activas por defecto; ?todas=1 muestra todas)
- POST   /api/categorias                              — Crea categoría nueva, valida nombre único
- PUT    /api/categorias/:id                          — Edita nombre, valida que no se duplique
- PATCH  /api/categorias/:id/activo                   — Activa o inactiva sin borrar
- POST   /api/productos/:id/categorias                — Asigna categoría a un producto
- DELETE /api/productos/:id/categorias/:id_categoria  — Remueve categoría de un producto

PBI-014 — Agregar productos al carrito por escaneo o búsqueda

Qué se implementó:
Se implementó la función agregarProducto() que maneja la lógica de agregar items al carrito,
incrementar cantidad si el producto ya existe, validar stock disponible y bloquear productos inactivos.
La búsqueda funciona por código de barras exacto primero, y luego por nombre con lista de sugerencias.

Archivos modificados:
- public/app.js
- backend/src/modules/productos/

Flujo de búsqueda:
- Se intenta buscar por código de barras exacto (/api/productos/barcode/:codigo)
- Si no encuentra, busca por nombre (/api/productos?q=)
- Si hay un solo resultado, se agrega directo al carrito
- Si hay varios, muestra lista de sugerencias para elegir

PBI-016 — Calcular subtotal, impuestos y total

Qué se implementó:
Se implementó el cálculo automático de importes dentro del módulo de ventas del POS.
El frontend recalcula dinámicamente subtotal, impuestos, descuento y total cada vez que
el cajero agrega, elimina o modifica productos en el carrito.

Archivos modificados:
- public/app.js
- backend/src/modules/pos/pos.model.js

Funcionalidades implementadas:
- Subtotal: suma de cantidad × precio unitario por cada partida
- Impuestos: tasa configurable aplicada sobre el subtotal
- Total: subtotal + impuestos - descuento
- Validación de importes no negativos en frontend y backend
- Normalización con parseFloat y Math.max(0, valor) antes de guardar en base de datos
- Redondeo a dos decimales con toFixed(2) para consistencia visual y contable
- Columnas subtotal, impuestos, descuento y total persistidas en tabla Venta

PBI-017 — Cobrar venta con método de pago

Qué se implementó:
Se desarrolló el flujo completo de cierre de venta dentro del POS.
El cajero puede seleccionar un método de pago válido antes de confirmar la operación.
Toda la operación se ejecuta dentro de una transacción MySQL para garantizar integridad de datos.

Archivos modificados:
- backend/src/modules/pos/pos.model.js

Funcionalidades implementadas:
- Métodos de pago disponibles: EFECTIVO, TARJETA, TRANSFERENCIA
- Transacción que incluye: INSERT Venta, INSERT Pago_Venta, INSERT Detalle_Venta,
  SELECT FOR UPDATE para validar stock, UPDATE stock en Producto,
  INSERT en Movimiento_Inventario
- Rollback automático si cualquier paso falla
- Validaciones: producto debe existir, cantidad mayor a cero, precio válido, stock suficiente
- Ventas guardadas automáticamente con estado COMPLETADA

PBI-020 — Cancelar venta y revertir inventario

Qué se implementó:
Se implementó el proceso de cancelación lógica de ventas manteniendo trazabilidad completa.
La operación completa se ejecuta dentro de una transacción.

Archivos modificados:
- public/app.js
- backend/src/api.js

Funcionalidades implementadas:
- Estado cambia a CANCELADA sin eliminar el registro original
- Restauración de stock: UPDATE Producto SET stock = stock + cantidad por cada partida
- Movimientos inversos de tipo ENTRADA en Movimiento_Inventario con referencia_tipo = "CANCELACION"
- Campos agregados en Venta: motivo_cancelacion y fecha_cancelacion
- Validaciones: solo ventas COMPLETADAS pueden cancelarse, no se puede cancelar dos veces,
  motivo de cancelación obligatorio
- Confirmación visual antes de cancelar desde el frontend
- Estado CANCELADA visible en tickets históricos y listado principal
